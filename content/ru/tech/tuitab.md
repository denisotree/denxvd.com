[tuitab](https://github.com/denisotree/tuitab) — это быстрый и мощный браузер табличных данных, который работает у вас в терминале. Написан на Rust. Он умеет работать с CSV, TSV, JSON, Parquet, Excel, SQLite и DuckDB из коробки (кроме того, по запросу я могу написать новые коннекторы) и позволяет фильтровать, сортировать, строить сводные таблицы, делать JOIN, считать новые столбцы и рисовать графики, не отрывая рук от клавиатуры. Я написал его в первую очередь для себя, потому что люблю анализ данных и люблю терминал. Мне нравится возможность быстро погрузиться в данные, посмотреть все интересующие сегменты, проверить простые гипотезы — и я не хочу для этого поднимать Jupyter или делать миллион SQL-запросов.

Я уже [рассказывал]({{< relref "visidata" >}}), что VisiData — один из моих самых любимых инструментов, и даже [сделал свой форк]({{< relref "visidata-ng" >}}), чтобы быстрее добавлять в него нужные мне фичи. Но в какой-то момент я упёрся в потолок и решил написать своё с нуля. В этой статье расскажу, почему так вышло, что tuitab умеет и как его можно легко попробовать.

---

### Почему я не остановился на форке VisiData

В форке [visidata-ng]({{< relref "visidata-ng" >}}) я прикрутил то, чего мне не хватало: подтверждение выхода, быстрый «процент от общего», форматирование валют и гибкие форматы дат. Но чем дольше я правил чужой код, тем яснее видел две вещи:

1. **Python-окружение — это боль.** Чтобы открыть Parquet, нужен `pyarrow`, для Excel — ещё пара пакетов, и всё это норовит поссориться между `pip`, `brew` и системным питоном. Я сам про эти [нюансы установки]({{< relref "visidata-install" >}}) писал — иногда настройка превращалась в отдельный квест.
2. **Скорость упиралась в архитектуру.** Хотелось колоночный движок под капотом, а не надстройки поверх готового приложения.

Поэтому tuitab я собрал на Rust поверх [Polars](https://www.pola.rs) — это дало и скорость, и один самодостаточный бинарь без единой внешней зависимости. То, что в форке было костылями, здесь стало частью ядра.

### Что умеет tuitab

1. **Шесть форматов из коробки**  
   CSV/TSV (разделитель определяется автоматически), JSON, Parquet, Excel, SQLite и DuckDB. Можно открыть один файл, пролистать целую папку или скормить данные через конвейер — без единого дополнительного пакета.

   ```bash
   # Открыть файл
   tuitab data.csv

   # Пролистать несколько файлов как список
   tuitab orders.csv customers.csv products.parquet

   # Прочитать данные из конвейера (формат задаётся флагом -t)
   psql -c "SELECT * FROM orders" --csv | tuitab -t csv
   ```

2. **Самодостаточный бинарь — никакого Python**  
   Никаких виртуальных окружений и конфликтов версий: ставится одной командой, а DuckDB и SQLite вкомпилированы прямо в бинарь. Вместе с `tuitab` появляются и короткие псевдонимы `ttab` и `ttb` — чтобы меньше печатать.

3. **Vim-навигация и закреплённые столбцы**  
   Перемещение на `h` `j` `k` `l`, в начало/конец — `gg` / `G`, постранично — `Ctrl+B` / `Ctrl+F`. Сортировка по текущему столбцу: `[` по возрастанию, `]` по убыванию, `r` — сброс. Нужный столбец можно закрепить слева по `!`, чтобы он не уезжал при горизонтальной прокрутке.

4. **Мгновенный анализ: статистика, частоты, графики**  
   Нажмите `I` — и получите сводку по столбцу (количество, пропуски, уникальные, минимум/максимум, среднее, медиана, мода, стандартное отклонение и квантили). `F` строит таблицу частот, а `V` рисует график прямо в терминале: для чисел — умную гистограмму (по правилу Фридмана — Дайкониса), для категорий — столбчатую диаграмму частот. Если закрепить столбец с датой или категорией, получаются линейные и сгруппированные графики.

   ![](/tech/tuitab/charts.gif)

5. **Сводные таблицы, JOIN и вычисляемые столбцы**  
   Это то, ради чего обычно открывают Python. В tuitab сводная строится так: закрепите столбцы для группировки, поставьте курсор на столбец, который нужно «развернуть», нажмите `W` и введите формулу агрегации, например `sum(revenue)`.

   ![](/tech/tuitab/pivot.gif)

   `J` запускает пошаговый мастер JOIN: выбираете второй файл (или уже открытый лист), тип `INNER` / `LEFT` / `RIGHT` / `OUTER` и ключевые столбцы.

   ![](/tech/tuitab/join.gif)

   А по `=` можно добавить вычисляемый столбец из выражения — поддерживаются арифметика, строковые и датовые функции, условия. Новый столбец появляется сразу рядом с курсором.

   ![](/tech/tuitab/compute.gif)

6. **Типы столбцов: валюта, проценты, даты**  
   По `t` столбцу можно задать тип: строка, целое, дробное, дата, дата-время, логический, проценты, валюта или размер файла. Для валюты сразу предложит `USD · EUR · GBP · JPY` и отформатирует значения нужным символом. Интересно, что именно эти вещи я когда-то руками добавлял в форк visidata, — здесь я сделал их готовыми из коробки, как и быстрый «процент от итога» по `zf`.

7. **Экспорт куда угодно и буфер обмена**  
   Сохранение по `Ctrl+S` — формат определяется по расширению (`.csv`, `.tsv`, `.parquet`, `.json`, `.xlsx`, `.sqlite`). А выделенные строки можно скопировать в буфер по `yr` сразу в TSV, CSV, JSON или Markdown — удобно, когда нужно быстро вставить аккуратную таблицу в документ или сообщение.

8. **ЙЦУКЕН тоже работает**  
   Раскладки кроме QWERTY (ЙЦУКЕН, QWERTZ, AZERTY) прозрачно ремапятся — горячие клавиши срабатывают независимо от того, на каком языке вы сейчас печатаете. Мелочь для большинства, но бесконечно удобно для тех, кто использует в работе как латиницу, так и кириллицу и постоянно переключается между ними.

### Как установить

Самый простой способ — через Cargo:

```bash
# Соберёт и установит tuitab (а заодно ttab и ttb)
cargo install tuitab
```

Поскольку DuckDB вкомпилирован из исходников, первая сборка занимает несколько минут — это нормально. Если ждать не хочется, есть готовые сборки.

```bash
# macOS / Linux через Homebrew
brew tap denisotree/tuitab
brew install tuitab
```

```bash
# Arch Linux — готовый бинарь из AUR
yay -S tuitab
```

```bash
# Debian / Ubuntu — скачайте .deb со страницы релизов и поставьте
sudo dpkg -i tuitab_*_amd64.deb
```

Готовые бинари под Linux и macOS (x86_64 / aarch64) лежат на [странице релизов](https://github.com/denisotree/tuitab/releases).

### Быстрый старт

- Откройте файл: `tuitab data.csv`, перемещайтесь по `h` `j` `k` `l`, прыжки — `gg` / `G`.
- Отсортируйте столбец: `[` по возрастанию, `]` по убыванию, `r` — сброс.
- Постройте график — `V`, статистика по столбцу — `I`, частоты — `F`.
- Выберите строки выражением: `|!=amount > 1000`.
- Добавьте столбец: `=`, затем, например, `revenue / units`.
- Сохраните результат — `Ctrl+S`, выход — `q`, справка в любой момент — `?`.

### Где tuitab пока не подходит

Я честно скажу: это не «убийца VisiData». VisiData старше, мощнее и оброс огромной экосистемой плагинов — если вам нужен редкий формат или хитрая интеграция, скорее всего, там она уже есть. tuitab я делал про другое: про скорость и нулевую возню с установкой.

Ещё один момент — как и VisiData, tuitab держит данные в памяти. На современной машине это комфортно для файлов в несколько гигабайт, но если у вас десятки гигабайт, то это уже задача для скриптов на [Polars](https://www.pola.rs) или [DuckDB](https://duckdb.org) напрямую. И проект ещё молодой (на момент статьи — версия `0.4.x`), так что местами что-то может вести себя не идеально.

### Планы и итог

tuitab уже стал моим основным инструментом для быстрых ad-hoc исследований — тем самым, что раньше делал в VisiData, только быстрее и без танцев с окружением.

Следующая большая штука, которую я хочу сделать, — это [MCP](https://modelcontextprotocol.io)-сервер для подключения к LLM. Идея в том, чтобы использовать tuitab как мощный движок анализа данных под управлением модели: LLM сама открывает файл, строит сводные, считает столбцы, фильтрует и читает результат, а вы просто описываете задачу словами. По сути, это превращает быстрый ручной анализ в терминале в разговор с данными на естественном языке — и при этом всё крутится на быстром локальном движке, без выгрузки данных в облако.

Проект открытый (Apache-2.0), так что любые идеи и баг-репорты приветствуются. В будущих статьях я разберу отдельные кейсы детальнее: сводные таблицы, JOIN и язык выражений на реальных данных.
